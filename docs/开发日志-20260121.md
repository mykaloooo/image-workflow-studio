#### 阶段4: 修复 API 调用错误 (18:20-18:30)

**问题**:
```
✗ module 'google.genai.types' has no attribute 'ImageConfig'
```

**问题分析**:
1. 初始代码使用了不存在的 `types.ImageConfig`
2. 使用了错误的 API 方法 `generate_image`
3. 参考原型程序发现应该使用 `generate_content`

**调试过程**:
1. 检查 `google-genai` 版本: 0.2.0
2. 列出 `types` 模块所有属性，确认没有 `ImageConfig`
3. 发现有 `GenerateImageConfig` 但不支持参考图
4. 查看原型程序源码，确认使用 `generate_content` + `GenerateContentConfig`

**解决方案**:
```python
# 修改前 (错误)
generate_content_config = types.GenerateContentConfig(
    image_config=types.ImageConfig(  # ❌ 不存在
        aspect_ratio=aspect_ratio,
        image_size=resolution,
    ),
)
response = self.client.models.generate_content(...)

# 修改后 (正确)
generate_content_config = types.GenerateContentConfig(
    temperature=1.0,
)
response = self.client.models.generate_content(
    model=self.model,
    contents=contents,  # 包含 prompt 和参考图
    config=generate_content_config,
)
```

**结果**: ✅ API 调用方法已修复，等待测试

ipcMain.handle('select-directory', async () => {

## 简化部署方案 (2026-01-21 深夜)

### 放弃 Electron 打包的原因
- 启动复杂，需要处理前后端进程协调
- 文件占用问题频繁
- 维护成本高
- 用户体验不佳

### 新方案：Flask 直接服务前端

#### 1. Flask 同时提供 API 和前端页面
```python
# 后端直接服务前端静态文件
FRONTEND_DIR = os.path.join(os.path.dirname(__file__), '..', 'frontend', 'dist')

@app.route("/")
def index():
    return send_file(os.path.join(FRONTEND_DIR, 'index.html'))

@app.route("/<path:filename>")
def serve_static(filename):
    return send_file(os.path.join(FRONTEND_DIR, filename))
```

## 技术说明：base64 图片存储

### 什么是 base64？
base64 是一种将二进制数据（图片、文件等）转换为 ASCII 字符串的编码方式。

**原理：**
- 图片是二进制数据（字节）
- base64 把每 3 个字节转换成 4 个 ASCII 字符
- 生成的字符串可以直接嵌入 JSON、HTML、JavaScript

**示例：**
```javascript
// 图片数据转 base64
const img_base64 = base64.b64encode(image_bytes).decode('utf-8')
// 结果: "iVBORw0KGgoAAAANSUhEUgAA..."

// 前端使用
<img src="data:image/png;base64,iVBORw0KGgoAAAANSUhEUgAA...">
```

### 本项目中的应用

**后端返回：**
```python
@app.route('/api/generate', methods=['POST'])
def generate_image():
    # 生成图片后，转换为 base64
    img_base64 = base64.b64encode(image_data).decode('utf-8')
    
    return jsonify({
        'data': f'data:image/png;base64,{img_base64}',  # 完整图片
        'thumbnail': f'data:image/jpeg;base64,{thumb_base64}',  # 缩略图
        'filename': 'generated_xxx.png'
    })
```

**前端存储：**
```javascript
// 保存到节点数据
setNodes(nds => nds.map(node => {
    if (node.id === id) {
        return {
            ...node,
            data: {
                ...node.data,
                imageUrl: result.images[0].data,  // base64 数据
                thumbnail: result.images[0].thumbnail,
                filename: result.images[0].filename,
                status: 'completed'
            }
        }
    }
    return node
}))
```

### 优点
1. **持久化** - 图片数据嵌入在项目 JSON 中，文件移走也能显示
2. **便携** - 项目文件包含所有图片，可直接分享
3. **无依赖** - 不依赖外部文件路径

### 缺点
1. **体积增加** - base64 比原始二进制大约大 33%
   - 1MB 图片 → 约 1.33MB base64 字符串
2. **内存占用** - 大量图片会占用较多内存
   - 100 张 1MB 图片 ≈ 133MB 数据

### 优化策略：缩略图方案
```python
# 后端同时生成缩略图
thumbnail_data = create_thumbnail(image_data, max_size=(200, 200))

# 前端显示优化
<img src={node.data.thumbnail || node.data.imageUrl} />  // 默认显示缩略图
// 点击查看完整图片时才加载 imageUrl（完整 base64）
```

### 使用场景
| 场景 | 建议 |
|------|------|
| 少量重要图片 | 完整 base64 |
| 大量图片 | 使用缩略图 |
| 最终交付 | 导出时重新生成完整文件 |

### 优化原理

#### v1: base64 方式
```
生成图片 → 转 base64 → 前端存储 → JSON 文件包含所有 base64
打开项目 → 加载所有 base64 → 内存占用大
```

#### v2: 路径方式 ⚡
```
生成图片 → 保存到 backend/outputs/ → 返回路径
前端存储 → 路径 (不是 base64)
打开项目 → 只加载路径，轻量
显示图片 → 浏览器请求 /api/images/xxx.png → 按需加载
```

## 内存泄漏修复 (2026-01-22)

### 问题描述

**现象**：应用运行约 15 分钟后浏览器崩溃，显示 `STATUS_BREAKPOINT` 错误。

**复现条件**：
- 打开应用后即使不操作
- 页面静置约 15 分钟
- 浏览器标签页崩溃

### 排查过程

#### 1. 检查事件监听器

对以下文件进行了事件监听器审查：

| 文件 | 检查项 | 结果 |
|------|--------|------|
| Canvas.jsx | paste 事件 (393-436行) | ✅ 有 cleanup |
| Canvas.jsx | click 事件 (832-836行) | ✅ 有 cleanup |
| ImageNode.jsx | mousedown 事件 (32-46行) | ✅ 有 cleanup |
| GenerationPanel.jsx | drag 事件 (66-74行) | ✅ 有 cleanup |
| GenerationPanel.jsx | paste 事件 (95-98行) | ✅ 有 cleanup |

**结论**：所有事件监听器都已正确清理，不是泄漏源。

#### 2. 发现真正的泄漏源

**日志数组无限增长** (`Canvas.jsx` 第43-46行)

```javascript
// 修复前 ❌
setLogs(prev => [...prev, { time, message, level }])
```

**问题**：日志永远不会被清理，15分钟内可能累积数百条日志，导致内存持续增长。

### 修复方案

**添加日志条数上限**（最多保留 100 条）：

```javascript
// 修复后 ✅
const MAX_LOGS = 100
const addLog = useCallback((message, level = 'info') => {
  const time = new Date().toLocaleTimeString('zh-CN', { hour12: false })
  setLogs(prev => {
    const newLogs = [...prev, { time, message, level }]
    // 超过最大条数时，删除旧日志
    if (newLogs.length > MAX_LOGS) {
      return newLogs.slice(-MAX_LOGS)
    }
    return newLogs
  })
}, [])
```

### 修改文件

| 文件 | 修改内容 |
|------|----------|
| `frontend/src/components/Canvas.jsx` | `addLog` 函数添加日志上限 |

### 其他观察

以下问题暂时保留观察，不是主要泄漏源：

1. **compressImage DOM 泄漏** - 临时创建的 Image/Canvas 对象，GC 会自动回收
2. **useEffect 循环更新** - 可能导致性能问题但不会累积内存

### 验证方法

如果问题仍然存在，可以使用 Chrome DevTools 进一步排查：

1. 打开 DevTools → Memory 标签
2. 拍摄两个堆快照（间隔 5-10 分钟）
3. 对比快照，查看哪些对象在增长
