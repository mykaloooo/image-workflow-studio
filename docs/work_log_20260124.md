# 📅 开发工作日志 - 2026-01-24

**项目名称**: Image Workflow Studio v2
**任务主题**: 适配 Antigravity/CherryStudio 中转协议，实现 4K 分辨率与多比例支持
**当前版本**: v3.0 (Backend Update)

---

## 1. 问题背景
在接入第三方 API 中转服务（基于 Antigravity 架构）时，遇到以下问题：
1.  **参数失效**：前端设置的 "4K分辨率" 和 "16:9" 比例无效，生成的图片始终为默认的 1K 正方形。
2.  **协议冲突**：在使用本地代理 (`127.0.0.1:8045`) 时，后端错误地切换到了 Google 原生协议，导致 `sk-` 开头的 Key 无法鉴权。
3.  **图片丢失**：API 虽然返回 200 成功，但后端无法提取图片地址，提示“未找到图片链接”。

## 2. 排查与分析
通过对比 CherryStudio 的成功请求截图和后端日志，分析出以下关键点：
*   **模型名参数传递**：该中转服务不通过标准的 API body 传递分辨率参数，而是要求将参数拼接在模型名称中（例如：`gemini-3-pro-image-4k-16x9`）。
*   **Base64 返回机制**：该 API 返回的不是 HTTP 图片链接，而是直接将图片的 Base64 数据嵌入在 Markdown 语法中（`![image](data:image/...)`）。
*   **逻辑漏洞**：`app.py` 中原本的逻辑是“检测到本地 IP 则强制使用 Google 协议”，这与本地运行的 OpenAI 兼容接口（Antigravity）冲突。

## 3. 实施变更

本次主要修改文件为 `backend/app.py`，核心变动如下：

### A. 协议判断逻辑优化
*   **修改前**：优先判断 URL 是否为本地地址，若是则强行使用 Google SDK。
*   **修改后**：优先判断 API Key 格式。只要 Key 以 `sk-` 开头，无论 URL 是远程还是本地，强制启用 **OpenAI 兼容模式**。

### B. 动态模型名称映射
*   新增了模型名重组逻辑。
*   在发送请求前，自动读取前端传入的 `resolution` (如 4K) 和 `aspect_ratio` (如 16:9)。
*   自动清洗原始模型名（去除 `-preview` 等），并拼接生成新的模型名：
    > 映射规则：`gemini-3-pro-image` + `4k` + `16x9` → `gemini-3-pro-image-4k-16x9`

### C. 响应解析增强
*   **支持 Base64**：引入了 `base64` 解码逻辑，专门处理 `data:image` 格式的内嵌图片。
*   **增强正则**：使用更通用的正则表达式 `!\[.*?\]\((.*?)\)` 提取 Markdown 图片内容。
*   **双重策略**：同时尝试提取 Markdown 图片和纯 HTTP 链接，确保兼容性。
